# Active Recall 仕様書

## MVPユーザーフロー

1. ユーザーがアプリを開く。
2. 保存済みの学習ソースを見る。
3. 本のソースを作成または選択する。
4. GPTが生成した学習内容をエントリーとして追加する。
5. ソース詳細画面からエントリーを見返す。
6. 必要に応じて内容を編集・削除する。

## 画面一覧

### ソース一覧

目的:

- すべての学習ソースを表示する。
- 新しいソース作成への入口を提供する。

状態:

- 空状態: 最初の本を追加できることを示す。
- 読み込み中: レイアウトが崩れないスケルトンまたは待機状態。
- エラー: 復旧可能なエラーメッセージ。
- データあり: タイトル、種別、更新日時を含むソース一覧。

### ソース詳細

目的:

- ソースのメタデータを表示する。
- そのソースに紐づく学習エントリーを表示する。
- 新規エントリー作成への入口を提供する。

状態:

- エントリーなし。
- エントリーあり。
- ソースが見つからない。
- データ読み込みエラー。

### ソースフォーム

目的:

- 学習ソースを作成・編集する。

項目:

- タイトル。
- ソース種別。
- 著者または作成者。
- 説明。
- 外部参照。

### エントリーフォーム

目的:

- GPTが生成した学習内容を作成・編集する。

項目:

- ソース。
- タイトル。
- 要約本文。
- キーポイント。
- 振り返り・リコール用プロンプト。

### エントリー詳細

目的:

- 1つのエントリーを集中して読める復習画面を提供する。

将来:

- リコール問題の練習。
- 復習履歴。
- AI生成の追加プロンプト。

## データモデル

### LearningSource

学習対象の元資料を表します。

必須フィールド:

- id。
- title。
- sourceType。
- createdAt。
- updatedAt。

任意フィールド:

- author。
- description。
- externalReference。
- 将来の認証用userId。

MVPで許可するソース種別:

- book。

将来のソース種別:

- article。
- video。
- lecture。
- podcast。
- course。

### LearningEntry

ソースから作成された学習単位を表します。

必須フィールド:

- id。
- sourceId。
- title。
- summary。
- createdAt。
- updatedAt。

任意フィールド:

- keyPoints。
- recallPrompt。
- reflection。
- 将来の認証用userId。

## Route Handlers

MVPでは、フォーム処理をサーバー側のアプリケーションフローで扱えるなら、公開APIルートは必須ではありません。

Route Handlersは次の場合にのみ導入します。

- Custom GPT Actionが構造化コンテンツをアプリへ送信する必要がある。
- 外部サービスが安定したHTTPエンドポイントを必要とする。
- ブラウザから呼ぶ専用エンドポイントに分けた方が責務が明確になる。

Route Handlersを導入する場合:

- 1つのhandlerは1つのリソースまたはアクションに絞る。
- リクエストボディをDB書き込み前に検証する。
- 成功・失敗レスポンスの形を統一する。
- DB内部の詳細を外へ出さない。
- 将来の認証と所有者チェックに備える。

将来候補:

- 外部ペイロードからソースを作成する。
- Custom GPTペイロードからエントリーを作成する。
- 連携テスト用ヘルスチェック。

## External API

MVPでの外部API方針:

- WebアプリからOpenAI APIを直接呼ばない。
- アプリ内GPTチャットを作らない。
- Custom GPTがアプリ外で構造化された学習内容を生成する。
- ユーザーがその内容を手動でアプリに保存する。

将来の外部API方針:

- Custom GPT ActionsがRoute Handlerを呼ぶ可能性がある。
- 問題生成や回答評価のためにOpenAI APIを導入する可能性がある。
- 記事、動画メタデータ、講義文字起こしの取り込みAPIを追加する可能性がある。

外部API追加時のルール:

- 提供元、目的、データ形状を文書化する。
- シークレットは環境変数にのみ保存する。
- タイムアウトと失敗時の処理を用意する。
- すべての入力データを検証・正規化する。
- 判断をADRに記録する。

## Validation Rules

ユーザーが制御できる入力は、保存前に必ず検証します。

LearningSource:

- タイトルは必須。
- ソース種別は必須。
- ソース種別はサポート済みの値のみ。
- URLとして扱う外部参照は有効な形式にする。
- 説明には適切な最大長を設ける。

LearningEntry:

- sourceIdは必須。
- タイトルは必須。
- 要約本文は必須。
- キーポイント、リコールプロンプト、振り返りは任意。
- コンテンツ項目にはDB保存とUI表示に適した最大長を設ける。

将来のCustom GPTペイロード:

- 未対応のソース種別は拒否する。
- 空の生成要約は拒否する。
- 配列、段落、Markdown風テキストを正規化する。
- GPT出力は信頼できない入力として扱う。

## UI挙動

- バリデーション失敗時は入力内容を保持する。
- 破壊的操作には確認を挟む。
- ナビゲーションは予測しやすく浅くする。
- 主要アクションは見つけやすく、視覚的には控えめにする。
- 空状態は次の行動を示すが、説明過多にしない。

## アクセシビリティ

- セマンティックな見出しを使う。
- フォームにはラベルとアクセシブルなエラー文を付ける。
- すべての操作をキーボードで扱えるようにする。
- 十分なコントラストを保つ。
- 色だけで状態を伝えない。
