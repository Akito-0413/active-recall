# Active Recall 開発ルール

## 最重要方針

AIエージェントはMVP境界を守ってください。

将来拡張を見越した設計は許可しますが、MVP外の機能を勝手に実装してはいけません。

## 実装前に読む順番

実装前に次の順番で読むこと。

1. `AGENTS.md`
2. `docs/vision.md`
3. `docs/requirements.md`
4. `docs/architecture.md`
5. `docs/specification.md`
6. `docs/decisions/` 配下の関連ADR
7. `node_modules/next/dist/docs/` 配下の関連Next.jsドキュメント

## Next.jsルール

このプロジェクトはNext.js 16.2.10を使用します。

古いNext.jsの記憶に基づいて実装してはいけません。

最新のNext.js 16とSupabaseの公式ドキュメントを参照し、推奨されていない設計やAPIは採用しないでください。

ルーティング、メタデータ、フォーム、サーバー処理、キャッシュ、Route Handlers、デプロイに関わる処理を書く前に、必ず `node_modules/next/dist/docs/` 配下の関連ガイドを確認してください。

Supabaseに関わる認証、DBクライアント、Row Level Security、マイグレーション、環境変数、サーバーサイド利用の設計を行う場合は、実装前にSupabase公式ドキュメントを確認してください。

## 実装規律

- 依頼されたタスクに必要な範囲だけを変更する。
- 既存パターンができた後は、それに従う。
- MVP中にOpenAI API呼び出しを追加しない。
- 汎用ノートアプリ化しない。
- ロードマップで必要になるまで復習スケジュールを追加しない。
- 主要な設計判断はADRに記録する。

## コンポーネントルール

- Server Componentsをデフォルトにする。
- Client Componentsは必要なブラウザ側インタラクションに限定する。
- フォームは小さく、アクセシブルにする。
- 機能固有コンポーネントは関連するルートや機能の近くに置く。
- 再利用が明確になるまで、大きな汎用抽象を作らない。

## データルール

- Supabase PostgreSQLを信頼できる保存先とする。
- DBアクセスは専用レイヤーに分離する。
- 書き込み前にバリデーションする。
- 将来のユーザー所有を意識する。
- 生のDBエラーをユーザーに表示しない。

## スタイリングルール

- Tailwind CSS 4を使う。
- UIはミニマルで学習に集中できるものにする。
- コア画面をマーケティングページ風にしない。
- 過剰なカード、装飾的グラデーション、高密度ダッシュボードを避ける。
- 読みやすいタイポグラフィと落ち着いた余白を優先する。

## 検証ルール

実装完了前に、原則として次を実行する。

- lint。
- 本番ビルド。
- 挙動が自明でない場合は焦点を絞ったテスト。
- UI変更時は主要フローの手動確認。

## ドキュメント更新ルール

次の場合はドキュメントを更新する。

- MVPスコープが変わる。
- データモデルが変わる。
- 外部APIが追加される。
- 認証の前提が変わる。
- 主要な技術・設計判断を行う。

将来のAIエージェントが同じ判断を繰り返さなくて済むように、重要な判断はADRに残してください。
