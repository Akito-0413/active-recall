# Active Recall 要件定義

## 1. 概要

Active Recallは、読書や学習内容をアクティブリコールによって定着させ、その結果を学習ログとして蓄積・管理するWebアプリケーションです。

ユーザーはCustom GPTと対話しながら学習内容を説明します。Custom GPTは深掘り質問を行い、内容を3から4個の要約ポイントに整理します。ユーザーが保存を承認すると、要約データがWebアプリに保存されます。

MVPでは読書を中心に扱いますが、将来的に動画、講義、記事などにも対応できる設計にします。

## 2. 目的

- 学習内容の記憶定着を促進する。
- アクティブリコールを習慣化する。
- 学びを構造化して蓄積する。
- 後から復習しやすい状態を作る。
- 会話全文ではなく、復習に必要な要約ログだけを保存する。

## 3. 対象ユーザー

- 読書・学習習慣がある人。
- 知識を効率的に定着させたい人。
- エンジニア、学生、研究者などの継続学習者。
- Custom GPTを学習補助に使いたい人。

## 4. 利用フロー

1. ユーザーが本を読む。
2. ユーザーがCustom GPTに内容を説明する。
3. Custom GPTが深掘り質問を行う。
4. Custom GPTが内容を3から4個の要約ポイントに整理する。
5. ユーザーが保存を承認する。
6. Custom GPT ActionsがWebアプリのRoute Handlerへ要約データを送信する。
7. Webアプリが学習ログをSupabase PostgreSQLに保存する。
8. ユーザーは後からログを閲覧・復習する。

手動登録フローもMVPに含めます。

## 5. ユーザーロール

MVPでは次の1種類を扱います。

- 学習者: Supabase Authでログインし、自分の学習ログだけを閲覧・作成・更新・削除できる体験。

ただし、Custom GPT Actionsからの保存APIはBearer APIキーで保護します。

将来的には次を追加できます。

- 管理者またはメンテナー。

## 6. 機能要件

### 6.1 学習ログ管理

- 学習ログを作成できる。
- 学習ログ一覧を表示できる。
- 学習ログ詳細を表示できる。
- 学習ログを削除できる。
- 学習ログの復習状態を更新できる。

### 6.2 Custom GPT連携

- Custom GPT Actionsから要約データを受け取れる。
- 受け取った要約データを検証できる。
- 要約ポイントは3から4件を基本とする。
- ユーザーが保存を承認した後にのみ保存APIを呼び出す。
- GPTとの会話全文は保存しない。

### 6.3 手動登録

- 本タイトルを入力できる。
- 要約ポイントを3から4件入力できる。
- 自分の気づきを入力できる。
- データ生成元を `manual` として保存できる。

### 6.4 復習機能

- 復習済みフラグを管理できる。
- 復習済みにした日時を保存できる。
- 未復習ログを抽出して表示できる。

### 6.5 認証とプロフィール

- メールアドレスとパスワードでサインアップできる。
- メールアドレスとパスワードでログインできる。
- Googleログインでアカウントを作成・ログインできる。
- ログアウトできる。
- ユーザー名をプロフィール画面で編集できる。
- 初期ユーザー名はメールアドレスの `@` より前を使う。
- 学習ログ画面はログイン済みユーザーのみ利用できる。

## 7. 保存データ

MVPの主データは学習ログです。

- 本タイトル。
- 要約ポイント3から4件。
- 自分の気づき。
- 作成日時。
- 更新日時。
- 復習状態。
- 復習済み日時。
- データ生成元。`manual` または `custom_gpt`。
- 所有者ユーザーID。手動作成ログはログイン中ユーザーに紐づく。

MVPではユーザープロフィールも保持します。

- ユーザーID。
- メールアドレス。
- ユーザー名。

将来拡張に備えて、ソース種別は設計上考慮します。MVPでは `book` を前提とします。

## 8. 非機能要件

- UIはシンプル、ミニマル、直感的であること。
- 学習に集中できる画面構成にすること。
- モバイル対応すること。
- 高速に一覧・詳細を表示できること。
- Custom GPT Actions向けAPIはBearer APIキーで保護すること。
- 会話全文を保存せず、DB負荷とプライバシーリスクを抑えること。
- データ保存はSupabase PostgreSQLを前提にすること。
- Supabase AuthのCookieベースSSR構成とRLSで、ユーザーごとのデータ分離を行うこと。
- 最新のNext.js 16とSupabaseの公式ドキュメントを参照し、推奨されていない設計やAPIは採用しないこと。
- Next.jsの実装では、必ずインストール済みドキュメントを確認すること。
- 主要フローには基本的なバリデーションとエラー表示を用意すること。

## 9. MVPスコープ

### 含む

- 学習ログCRUD。
- メール+パスワードログイン。
- Googleログイン。
- アカウント画面でのユーザー名更新。
- 手動登録。
- Custom GPT Actionsからの要約保存。
- 学習ログ一覧画面。
- 学習ログ詳細画面。
- 未復習ログ一覧。
- 復習済みフラグ更新。
- Bearer APIキーによるGPT連携API保護。

### 含まない

- アプリ内OpenAI API呼び出し。
- アプリ内GPTチャット。
- クイズ生成。
- AIによる回答評価。
- 復習通知。
- 間隔反復スケジューリング。
- 学習分析。
- 会話全文保存。
- 動画、講義、記事、PDFの自動取り込み。

- Row Level Security。
- AIによるリコール問題生成。
- AIによる回答評価。
- 間隔反復スケジューリング。
- 復習通知。
- 学習分析。
- 動画、講義、記事への対応。
- ソースやログ横断の検索・フィルタ。
- エクスポートとインポート。

## 11. 受け入れ条件

MVP完了の条件は次の通りです。

- 学習者が手動で学習ログを作成できる。
- Custom GPT Actionsから学習ログを作成できる。
- 学習者がログ一覧と詳細を閲覧できる。
- 学習者がログを削除できる。
- 学習者が未復習ログを確認し、復習済みにできる。
- 学習者がメール+パスワードまたはGoogleでログインできる。
- 学習者が自分のユーザー名を更新できる。
- 未ログイン状態では学習ログ画面に入れない。
- ログイン中の学習者は自分のログだけを見られる。
- GPT連携APIがBearer APIキーで保護されている。
- 会話全文を保存せず、要約ポイントと気づきのみ保存する。
- 主要画面に空状態、読み込み、エラー、成功状態がある。
- lintと本番ビルドが通る。
