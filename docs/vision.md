# Active Recall ビジョン

## プロダクト

Active Recall は、主に読書から得た学習内容を保存し、振り返り、記憶に残すための学習Webアプリです。

MVPでは読書を中心に扱いますが、将来的に動画、講義、記事などにも対応できるデータ設計にします。

## 対象ユーザー

- 読書内容を長期的に理解・記憶したい個人学習者。
- 本から得た知識を仕事や研究に活かしたい知識労働者。
- Custom GPTで要約や学習メモを作成しているユーザー。

## 解決する課題

読書メモは、作っただけで見返されない受動的な保管庫になりがちです。

Active Recall は、読書内容を「思い出すための学習単位」として保存し、あとから復習しやすくすることを目的とします。

## プロダクト原則

- 学習優先: すべての画面は、ユーザーが思い出す・振り返る・理解を深める行為に集中できるようにする。
- ミニマル: 情報量の多すぎるダッシュボードや装飾的なUIを避ける。
- 出典を保つ: 学習内容は、元になった本や資料と紐づける。
- AI支援だがAI依存にしない: MVPではCustom GPTが学習内容の作成を支援し、Webアプリは保存と復習を担当する。
- 拡張可能にする: 将来的な問題生成、復習スケジュール、複数ソース種別に備える。

## MVPの成功状態

MVPは、ユーザーが次のことをできる状態を成功とします。

- GPTが生成した読書要約をアプリに保存できる。
- 保存した学習内容を本ごとに整理できる。
- シンプルで集中しやすいUIで学習内容を見返せる。
- 将来の復習・問題生成に必要なメタデータを保持できる。

## MVPでやらないこと

- 複数ユーザーや共同編集。
- アプリ内GPTチャット。
- 自動問題生成。
- 復習スケジュール。
- PDF、動画、記事の自動取り込み。
- 複雑なナレッジグラフ表示。

## AIエージェント向け判断基準

実装判断に迷った場合は、次を優先してください。

- 機能を増やすより、MVPの中核フローを完成させる。
- 巧妙なUIより、明確なデータ境界を優先する。
- 可能な限りサーバー側でデータアクセスする。
- 見た目の新規性より、読みやすさとアクセシビリティを優先する。
- Supabase AuthとRLSで、自分の学習ログだけを安全に扱える構造にする。
