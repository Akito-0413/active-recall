# Active Recall ロードマップ

## Phase 0: 設計基盤

目標:

- プロダクトビジョンを定義する。
- MVP境界を明確にする。
- アーキテクチャと実装ルールを文書化する。
- ADR運用を開始する。

完了条件:

- 中核ドキュメントが揃っている。
- 初期の技術判断が記録されている。
- AIエージェントがプロダクト方針を推測せずに実装へ進める。

## Phase 1: MVP

目標:

- GPT生成の要約ポイントを学習ログとして保存する。
- Custom GPT Actionsから学習ログを保存する。
- 未復習ログを表示し、復習済みフラグを更新する。
- ミニマルなUIで保存済みログを見返す。
- Supabase PostgreSQLに永続化する。

明示的に除外するもの:

- 認証。
- アプリ内OpenAI API呼び出し。
- 自動問題生成。
- 間隔反復。
- 記事、動画、講義の取り込み。

完了条件:

- 学習ログの主要CRUDフローが動く。
- 将来の認証付き所有に対応できるデータモデルになっている。
- 空状態、読み込み、エラー、成功状態がある。
- lintとbuildが通る。

## Phase 2: 学習体験の改善

目標:

- 集中して復習できる画面を改善する。
- 検索とフィルタを追加する。
- 必要性が明確なら軽量なタグや分類を追加する。
- Custom GPTからの受け渡し手順を改善する。

完了条件:

- ユーザーが過去の学習ログを素早く探せる。
- 復習体験が管理作業ではなく学習に近づく。

## Phase 3: AIリコール機能

目標:

- 保存済みログからリコール問題を生成する。
- 回答へのフィードバックを生成する。
- 追加の問いかけを提案する。
- アプリ内OpenAI API連携を検討する。

完了条件:

- AI生成コンテンツが安全に検証・保存される。
- ユーザーが元の要約と生成されたリコール素材を区別できる。
- 外部API判断がADRに記録されている。

## Phase 4: 復習スケジューリング

目標:

- 間隔反復の概念を導入する。
- 復習セッションを記録する。
- 次回復習日を提案する。
- 期限切れまたは優先度の高いログを表示する。

完了条件:

- UIを複雑にしすぎず、復習スケジュールが学習を助ける。
- スケジューリングのデータモデルが文書化・テストされている。

## Phase 5: ソース種別の拡張

目標:

- 記事、動画、講義、ポッドキャスト、コースに対応する。
- ソース種別ごとのメタデータを追加する。
- インポートや文字起こしワークフローを検討する。

完了条件:

- 新しいソース種別が、可能な限り既存の学習ログモデルを再利用している。
- ソース固有メタデータが本ベースのフローを複雑にしていない。

## Phase 6: 認証とパーソナライズ

目標:

- Supabase Authを追加する。
- データをユーザーごとに分離する。
- Row Level Securityを導入する。
- ユーザー設定を追加する。

完了条件:

- 匿名MVPの前提が安全に置き換えられている。
- Data Access層でユーザー所有を強制している。
- 認証関連の判断がADRに記録されている。
